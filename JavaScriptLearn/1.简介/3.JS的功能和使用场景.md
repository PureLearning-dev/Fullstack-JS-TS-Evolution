# JavaScript 的功能和使用场景

## 前言：学了一门语言，到底能干啥？

学任何一门编程语言之前，你大概都会问自己：**"学了它，我能做啥？"** 如果你学的是 C，可能是为了写嵌入式系统或操作系统。学的是 Java，可能是为了进大厂做后端。学的是 Python，可能是为了搞 AI、数据分析或者写脚本处理 Excel。

那 JavaScript 呢？

答案是：**几乎所有你能想到的软件领域，JavaScript 都有涉足。** 它就像编程语言界的"瑞士军刀"——切开包装用主刀，开红酒用钻子，拆快递用剪刀。虽然每一样功能可能都不是业界最强，但架不住它啥都行。

下面我们就来逐一拆解 JavaScript 的能力版图。我会尽量把每一个场景讲清楚，包括它适合干什么、不适合干什么、用什么工具和框架。看完这一篇，你就知道学完 JS 之后能去做什么了。

## 一、前端开发 —— JavaScript 的龙兴之地

这当然是 JavaScript 最核心、最传统的战场。浏览器是 JS 的"故乡"，一切故事的起点。可以说，JavaScript 就是为了让网页"动起来"而生的。

### 1.1 DOM 操作 —— 网页的骨架操控术

**DOM**（Document Object Model，文档对象模型）是浏览器给网页建立的一个树状结构模型。网页上的每一个元素——标题 `<h1>`、段落 `<p>`、按钮 `<button>`、图片 `<img>`——在这个模型里都是一个"节点"。JavaScript 可以通过 DOM API 对这些节点做任何事情：

- **查找元素**：`document.querySelector('.my-class')`、`getElementById('myId')`
- **修改内容**：`element.textContent = '新的文字'`、`element.innerHTML = '<strong>粗体</strong>'`
- **修改样式**：`element.style.color = 'red'`
- **创建/删除元素**：`document.createElement('div')`、`element.remove()`
- **修改属性**：`element.setAttribute('src', 'image.jpg')`

一句话概括：**有了 DOM 操作能力，JS 可以随时随意地修改页面上的任何东西。** 网页不再是写死的静态 HTML 文档，而是一块可以实时编辑的画布。

这在现代前端框架（React、Vue 等）里已经被高度抽象化了——你通常不需要直接操作 DOM，框架会帮你高效地完成。但理解 DOM 是理解一切前端技术的基石。就好比你学车不一定需要会修发动机，但你至少要知道方向盘和油门是干什么的。

### 1.2 事件处理 —— 让网页能"互动"

静态的网页像一张海报，再漂亮也只能看不能碰。而 JavaScript 的**事件系统**让网页变得可以交互——用户的每一次点击、每一次键盘按下、每一次鼠标移动、每一次触摸屏幕，都会触发"事件"，JS 可以监听这些事件并做出响应。

常用的事件类型：

| 事件类型                      | 触发时机    | 典型用途         |
|---------------------------|---------|--------------|
| `click`                   | 鼠标点击    | 按钮点击、链接跳转    |
| `submit`                  | 表单提交    | 验证用户输入、发送数据  |
| `keydown` / `keyup`       | 键盘按下/释放 | 快捷键、搜索联想     |
| `mousemove`               | 鼠标移动    | 拖拽、悬停效果、画布绘图 |
| `scroll`                  | 页面滚动    | 无限加载、导航栏吸顶   |
| `resize`                  | 窗口大小变化  | 响应式布局调整      |
| `touchstart` / `touchend` | 手机触摸    | 移动端手势交互      |
| `input` / `change`        | 表单输入变化  | 实时搜索、即时校验    |

配合事件对象（`event`），你还能获取到更细的信息：鼠标的坐标位置、按下了哪个键、用户在文本框里输入了什么内容。通过**事件冒泡**和**事件委托**机制，你甚至可以用一个事件处理器管理上百个子元素——这是前端性能优化的一项基本功。

没有事件，JS 就只是网页渲染完就傻站着的一个旁观者；有了事件，JS 变成了全程参与互动的游戏玩家。

### 1.3 CSS 操作 —— 让网页"变装"

JavaScript 不仅能改 HTML 的内容，还能动态修改 CSS 样式。这意味着网页的外观可以根据用户的行为实时变化：

- **切换主题**：白天模式/黑夜模式，一键换肤
- **动画效果**：淡入淡出、滑入滑出、弹跳、旋转
- **交互反馈**：鼠标悬停放大、按钮按下凹陷效果、加载时骨架屏过渡
- **响应式变化**：根据窗口大小动态调整布局
- **滚动视差**：不同层级的元素以不同速度滚动

通过 `element.style` 修改内联样式，通过 `element.classList` 切换 CSS 类，通过 CSSOM（CSS 对象模型）操作样式表本身，JS 提供了多种层次的方式来控制页面颜值。

但真实项目中，大量的动画和过渡效果我们会优先用纯 CSS（`transition`、`animation`、`@keyframes`）来做——因为浏览器对 CSS 动画有 GPU 加速优化。JS 操作样式主要用于"CSS 做不到"的场景，比如拖拽跟随、动态计算布局等。这个取舍是前端性能的基本常识。

### 1.4 浏览器 API —— 藏在浏览器里的百宝箱

现代浏览器除了能显示网页，还提供了一套丰富的 **Web API**。JS 可以直接调用这些 API，让网页应用拥有接近原生应用的能力。以下是你必须了解的几个大类别：

**网络请求**
- `fetch()`：现代的网络请求方式，基于 Promise，支持 GET/POST/PUT/DELETE、文件上传、流式读取等。已经取代了老旧的 `XMLHttpRequest`。
- `WebSocket`：浏览器和服务器之间建立长连接，实现实时双向通信（聊天室、实时股价、在线协作、游戏）。
- `EventSource`（Server-Sent Events / SSE）：服务器向浏览器单向推送事件流（AI 对话的流式输出很多就用这个）。
- `WebRTC`：浏览器之间直接的点对点通信（视频通话、屏幕共享、P2P 文件传输）。

**本地存储**
- `localStorage`：持久化的本地键值存储，数据不会随浏览器关闭而消失，容量约 5-10MB。
- `sessionStorage`：会话级别的存储，关闭标签页就清空，适合存临时的表单草稿等。
- `IndexedDB`：浏览器里的事务型 NoSQL 数据库，可以存大量结构化数据（几百 MB 甚至 GB 级）。适合离线应用、缓存大量 API 数据。
- `Cache API`：配合 Service Worker 实现精细化的 HTTP 缓存控制，是 PWA 离线能力的核心。
- `Cookies`（`document.cookie`）：最古老的客户端存储方式，每次 HTTP 请求会自动携带，主要用于身份认证 token 的存储（不过现在更推荐用 `HttpOnly` Cookie 由服务端管理）。

**多媒体**
- `Audio API`（Web Audio API）：创建、处理、合成音频。可以做音乐可视化、在线 DAW（数字音频工作站）、游戏音效引擎。
- `Video API`：控制 `<video>` 元素的播放、暂停、进度、画质切换等。
- `MediaDevices.getUserMedia()`：访问用户的摄像头和麦克风（视频通话、拍照、录音）。
- `Canvas API`：2D 绘图能力，可以画图形、渲染图表、处理图片像素、做游戏画面。
- `WebGL` / `WebGL 2`：3D 图形渲染，基于 OpenGL ES，可以在浏览器里跑 3D 游戏、产品展示、数据可视化。
- `WebGPU`：下一代的图形和计算 API，比 WebGL 更底层、更高性能、更现代，支持 GPU 通用计算。

**设备能力访问**
- `Geolocation API`：获取用户地理位置（经度、纬度、海拔、速度）。
- `Clipboard API`：读写系统剪贴板（复制/粘贴功能）。
- `Notification API`：发送系统通知（需要用户授权）。
- `Vibration API`：让手机振动（移动端网页也可以振！）。
- `Battery Status API`：查询设备电量（不过出于隐私考虑，各浏览器的支持不一致）。
- `Web Bluetooth API`：和蓝牙设备通信。
- `Web USB API`：通过 USB 连接外部设备。
- `Screen Wake Lock API`：防止屏幕自动休眠。

**性能与观测**
- `Performance API`：精确测量页面加载时间、资源加载耗时、JS 执行性能等。
- `Intersection Observer`：观察元素是否进入视口（懒加载图片、曝光埋点、无限滚动）。
- `Mutation Observer`：观察 DOM 树的变化。
- `Resize Observer`：观察元素尺寸的变化（比监听 window resize 更精确）。

这些 API 的存在意味着：**一个现代浏览器，本质上就是一个迷你操作系统。** JS 是这个操作系统的主要编程语言。这也是为什么有人开玩笑说 Chrome 是"世界上安装量最大的 JS 运行环境"。

### 1.5 单页应用（SPA）—— 像桌面软件一样的网页

SPA（Single Page Application，单页应用）是现代前端开发的主流范式。它的理念是：**整个应用只加载一个 HTML 页面，之后所有的页面切换和内容更新都由 JavaScript 动态完成，不再向服务器请求新的 HTML 文件。**

这意味着什么？你打开 Gmail、飞书文档、B 站网页版——在页面之间切换时不会看到浏览器"刷新"的闪白，体验平滑得像桌面软件一样。

SPA 的核心技术：
- **前端路由**：通过 `history.pushState` API（或 `hashchange` 事件）改变浏览器地址栏的 URL，但不触发页面刷新。由 JS 拦截 URL 变化，渲染对应的"虚拟页面"。
- **状态管理**：整个应用的状态（用户信息、表单数据、加载状态等）集中管理，任何组件都可以读取和修改。
- **按需加载（Lazy Loading / Code Splitting）**：首屏只加载必需代码，其他页面按需加载，减少初始加载时间。

主流框架：
- **React**（Meta）—— 生态最庞大，岗位需求最多。配套 Next.js 做 SSR/SSG，React Native 做移动端。
- **Vue**（尤雨溪）—— 上手最友好，渐进式设计，中文生态极强。配套 Nuxt.js 做全栈，Vite 做构建。
- **Angular**（Google）—— 企业级"全家桶"方案，自带路由、HTTP、表单、依赖注入，TypeScript 原生支持。
- **Svelte**（Rich Harris）—— 编译时框架，没有 Virtual DOM，构建产物极小，理念超前。
- **SolidJS**（Ryan Carniato）—— 和 React 相似的 API，但响应式粒度极细，性能接近原生 JS。

你不需要全部学会——选一个深耕就好。但背后的核心概念（组件化、状态管理、路由、生命周期）是所有框架通用的。

### 1.6 SSR / SSG —— 服务端渲染，更快更强

虽然 SPA 体验很好，但它有一个天生的弱点：**首屏加载慢，且对 SEO 不友好。** 因为 SPA 的 HTML 基本是一个空壳，所有内容和逻辑都由 JS 在浏览器里渲染，搜索引擎的爬虫不一定能正确地等待和执行你的 JS（虽然 Google 声称可以，但效果不稳定），很多国内搜索引擎更是一行 JS 都不跑。

于是就出现了服务端渲染（Server-Side Rendering, SSR）和静态站点生成（Static Site Generation, SSG）：

- **SSR**：每次请求时，在服务器上把 React/Vue 组件渲染成完整的 HTML 字符串，发送给浏览器。浏览器收到的是已经包含内容的 HTML（对 SEO 友好），之后 JS 接管交互（这个过程叫"水合" hydration）。
- **SSG**：构建时将全站预渲染为静态 HTML 文件，部署到 CDN 上。速度极快，适合博客、文档站、营销页面等内容变化不太频繁的场景。
- **ISR（Incremental Static Regeneration）**：相当于 SSG + 按需更新。大部分页面是静态的，但可以指定某些页面每隔一段时间自动在服务端重新生成一次。

代表框架：
- **Next.js**（React 生态，Vercel 维护）—— 全栈框架的标杆，支持 SSR、SSG、ISR、Server Components、API Routes。
- **Nuxt.js**（Vue 生态）—— Vue 版的 Next.js，同样功能完备。
- **Remix**（React 生态，Shopify 收购）—— 主打 Web 标准，更贴近浏览器原生 API。
- **Astro**（框架无关）—— 主打"岛屿架构"，默认零 JS，只在需要交互的组件上加载 JS。适合内容型网站。

### 1.7 PWA —— 把网页变成"App"

PWA（Progressive Web Application，渐进式 Web 应用）是一组技术与理念的集合，目标是把网页应用改造得接近原生 App 的体验：

- **Service Worker**：在浏览器后台运行的 JS 脚本，可以拦截网络请求、管理缓存、接收推送消息。它是 PWA 离线能力的核心。
- **Web App Manifest**（`manifest.json`）：定义应用名称、图标、启动画面、全屏模式等。当用户把 PWA"添加到主屏幕"后，它可以像原生 App 一样没有浏览器地址栏地运行。
- **离线能力**：网络断了也能打开应用、浏览已缓存的内容。你在地铁里信号不好时还能刷之前加载过的页面，就是 Service Worker 的功劳。
- **Push Notification**：即使浏览器关闭，也能接收到服务器推送的通知。
- **Background Sync**：在网络恢复时自动同步离线时的操作（比如离线时写的评论，有网了自动提交）。

像 Twitter/X、Pinterest、Starbucks、Uber 都提供了 PWA 版本。对于不想开发两个原生 App 但又需要离线能力和安装体验的场景，PWA 是一个高性价比的折中方案。

## 二、后端开发 —— JavaScript 走出浏览器

如果说前端是 JS 的老家，那后端就是它征服的第一块新领地。2009 年 Node.js 的诞生改变了一切，而今天这个领域已经是三雄争霸的局面。

### 2.1 Node.js —— 让 JS 跑在服务器上

**核心能力**：Node.js = Chrome 的 V8 引擎 + 操作系统能力

V8 引擎本身只负责执行 JavaScript 代码，它不关心自己跑在浏览器里还是服务器上。Node.js 做的事情就是——把 V8 从浏览器里拿出来，给它装上操作系统的能力：

- **文件系统（`fs` 模块）**：读写文件、创建目录、监听文件变化。你可以用 JS 写一个批量重命名脚本，或者处理用户上传的文件。
- **网络（`http`/`https`/`net`/`dgram` 模块）**：创建 HTTP 服务器、发送 HTTP 请求、建立 TCP/UDP 连接。最直观的——几行代码起一个 Web 服务器。
- **进程管理（`child_process`/`cluster`/`worker_threads` 模块）**：启动子进程、多核 CPU 并行、多线程计算。
- **加密（`crypto` 模块）**：哈希、加密解密、生成随机数、数字签名。
- **流（`stream` 模块）**：处理大文件的增量和管道传输，不用把整个文件加载到内存。
- **路径处理（`path` 模块）**、**操作系统信息（`os` 模块）**、**事件（`events` 模块）** 等等。

**为什么 Node.js 适合做后端**：事件驱动 + 非阻塞 I/O 模型。Node.js 是单线程的（主线程只有一个），但所有的 I/O 操作（读文件、查数据库、调外部 API）都是异步非阻塞的——发出请求后就干别的事了，I/O 完成时通过回调/Promise/事件通知你处理结果。这使得 Node.js 可以将资源集中在处理高并发 I/O 上，而不需要为每个连接开一个线程。当你的应用瓶颈在 I/O（绝大多数 Web 应用都是）而非 CPU 密集计算时，Node.js 的表现非常优秀。

**典型的 Node.js 后端技术栈**：
- **Web 框架**：Express（最经典最简洁）、Fastify（高性能）、Koa（Express 团队的升级版）、Hono（新秀，性能极佳，支持多运行时）
- **全栈框架**：Next.js（React）、Nuxt.js（Vue）—— 前后端在同一个项目里，API Routes 直接写后端逻辑
- **ORM / 数据库工具**：Prisma（类型安全的 ORM，当前最火）、Drizzle ORM（轻量级）、TypeORM、Sequelize
- **数据库**：PostgreSQL、MySQL、MongoDB、Redis、SQLite
- **实时通信**：Socket.IO、ws
- **认证**：NextAuth.js（Auth.js）、Passport.js、Lucia Auth
- **GraphQL**：Apollo Server、GraphQL Yoga、Mercurius

**真实世界中的 Node.js 用户**：Netflix（减少了 70% 的启动时间）、PayPal（后端全面迁移到 Node.js 后，代码量减少 33%，响应时间减少 35%）、LinkedIn（移动端后端从 Ruby on Rails 迁移到 Node.js，服务器从 30 台降到 3 台）、Uber、eBay、Walmart、NASA……这个名单还能列很长。

### 2.2 Deno —— Node.js 的灵魂续作

2018 年，Node.js 的创始人 **Ryan Dahl** 在 JSConf EU 上做了一个著名的演讲《**10 Things I Regret About Node.js**》，列举了他对 Node.js 的十大遗憾：

1. 没有坚持使用 Promise（导致 callback hell 的生态惯性）
2. 安全性不够（默认给了太多系统权限）
3. 构建系统 GYP 太复杂
4. `package.json` 的设计问题
5. `node_modules` 的"黑洞"式依赖管理
6. `require` 省略 `.js` 扩展名
7. `index.js` 的隐含默认行为
8. 没有原生 TypeScript 支持
9. 模块系统（CommonJS）和浏览器标准（ESM）不一致
10. 没有标准化的 lint 和 formatter

然后他宣布：**我正在做一个新东西来解决这些问题。那就是 Deno。**

**Deno 的核心特点**：
- **原生 TypeScript 支持**：`deno run app.ts`，不需要任何配置，直接跑 TS 文件。你甚至不需要装 TypeScript 编译器——Deno 内置了。
- **安全优先**：默认情况下，Deno 脚本没有文件系统、网络、环境变量的访问权限。需要什么权限，你需要显式地通过命令行参数授权：`deno run --allow-read --allow-net app.ts`。这样即使运行了一个恶意脚本，它也没法偷偷读你的文件。
- **去中心化的模块管理**：没有 `package.json`，没有 `node_modules`。依赖通过 URL 直接导入：`import { serve } from "https://deno.land/std/http/server.ts"`。Deno 会缓存下载的文件，版本通过 URL 来控制。这个设计比较激进，有利有弊——依赖管理和可复现性方面存在争议，但在理念上是对 `node_modules` 的一次大胆的简化尝试。
- **内置开发工具链**：内置了代码格式化工具 `deno fmt`、代码检查工具 `deno lint`、测试运行器 `deno test`、打包器、文档生成器、依赖检查器……不用像 Node.js 项目那样安装 ESLint + Prettier + Jest + 十几项配置文件。
- **Web 标准优先**：尽可能使用浏览器标准的 API。比如 `fetch` 就是浏览器的 `fetch`，不需要安装 `node-fetch`；`Request` 和 `Response` 对象遵循 Web 标准。

到 2025 年，Deno 已经比较成熟，生态逐渐丰富。但它能不能全面取代 Node.js，仍然不好说——Node.js 庞大的 NPM 生态不是那么容易撼动的。好消息是 Deno 从 2.x 版本开始兼容了大量 NPM 包（通过 `npm:` 前缀导入），这让迁移成本大幅降低。

### 2.3 Bun —— 性能怪兽

2022 年，Jarred Sumner 发布了 **Bun**。Bun 的目标非常直接：**我要比 Node.js 和 Deno 都快。**

Bun 使用 **Zig** 语言编写，内置的 JS 引擎是 **JavaScriptCore**（Safari 的 JS 引擎，Jarred 之前在 Stripe 工作时对 JSC 有深入优化经验），而不是 V8。核心卖点就是速度：

- **运行时**：JS 执行速度与 V8 在不同使用场景下有来有回，启动速度更快。
- **包管理器**：`bun install` 比 `npm install` 快很多（官方 benchmark 宣称快 25 倍以上）。
- **打包器**：内置打包和转译能力，不需要 Webpack/Vite。
- **测试运行器**：内置测试框架，和 Jest 兼容的 API。
- **原生 TypeScript 和 JSX 支持**：不需要任何配置，直接跑 `.ts` 和 `.tsx` 文件。
- **兼容 Node.js**：大部分 Node.js API 和环境可以直接使用，迁移成本低。

Bun 在 2023 年 9 月发布了 1.0 版本，目前发展势头不错。对于新项目追求极致性能的场景（尤其是开发体验的响应速度），Bun 是一个令人兴奋的选择。

### 2.4 后端可以做什么？具体场景列举

- **RESTful API 服务**：最经典的用法。用 Express/Fastify/Hono 写接口，连数据库，返回 JSON。
- **实时应用**：WebSocket 连接让服务器可以主动推送消息给客户端——在线聊天、协同编辑、股票行情、多人在线游戏、直播弹幕。
- **GraphQL API**：前端可以精确描述自己需要什么数据，后端按需返回，避免过度/不足获取。
- **微服务**：Node.js 轻量、启动快，非常适合微服务架构。配合 Docker 和 Kubernetes，可以弹性伸缩。
- **BFF（Backend For Frontend）**：专门为前端设计一个中间层服务，聚合多个后端 API 的数据，格式化成前端最方便使用的结构。Node.js 做 BFF 层非常常见。
- **SSR 服务**：用 Next.js/Nuxt.js 在服务器上渲染页面。
- **Serverless / Edge Functions**：在 AWS Lambda、Vercel Edge、Cloudflare Workers 等平台上按需运行的函数。JS/TS 是这些平台的一等公民——冷启动速度极快，特别适合 Serverless 场景。
- **CLI 工具**：用 JS 写命令行工具非常方便——`npm create vite`、`eslint`、`prettier`、`git-cz`，包括你用来管理项目的 `git`……都是 JS 写的。好吧，`git` 不是。但很多前端工具链 CLI 都是。

## 三、移动端开发 —— 一份代码，两个平台

移动端原生开发意味着——iOS 要用 Swift/Objective-C 写一套，Android 要用 Kotlin/Java 写另一套。两套代码、两个团队、两个发布周期。对小团队或初创公司来说，成本是巨大的。

JS 生态提供的解决方案是：**跨平台框架**，用 Web 技术栈写移动 App。

### 3.1 React Native —— Meta 出品的跨平台利器

**React Native（RN）** 是目前最流行的 JS 移动端开发框架。它的核心机制是：

- 你的 JS 代码跑在一个独立线程里（JavaScriptCore 引擎）
- JS 线程和原生 UI 线程通过一个 **Bridge**（桥）进行异步通信
- JS 这边告诉原生那边"在屏幕坐标 (x, y) 处渲染一个红色按钮"，原生那边就用真正的原生组件去渲染——不是 WebView 模拟的，是真实的原生 UI 组件

所以 RN 应用的用户体验（滚动、手势、动画）明显优于 WebView 套壳方案。你用 JS 写逻辑，而用户看到的是原生按钮、原生列表、原生导航栏。

RN 支持热更新（CodePush/Expo Updates），可以绕过 App Store 审核直接推送 JS 代码更新。

**Expo** 是 RN 之上的一个工具平台，进一步降低了开发门槛。有了 Expo，你可以：
- 不需要安装 Xcode 或 Android Studio 就能开发（通过 Expo Go App 在真机上实时预览）
- 不需要自己管理原生构建配置（Expo 托管了打包和签名）
- 使用 Expo SDK 访问原生的相机、传感器、地理位置等

Instagram、Facebook（部分）、Shopify、Walmart、Bloomberg 的移动端 App 都使用了 React Native。

**适用场景**：UI 不算极其复杂、不需要大量原生级动画的应用——社交、电商、新闻、工具类应用。**不适用场景**：需要大量 3D 渲染、复杂视频处理或极致性能的高帧率应用（如大型 3D 游戏、高性能视频编辑等）。

### 3.2 Ionic & Capacitor —— Web 技术栈 + 原生壳

**Ionic** 的思路和 RN 不同：不追求原生组件的渲染，而是直接在 WebView 里渲染 HTML/CSS/JS 界面。

这听起来很"退步"，但实际上现代手机的 WebView 性能已经足够好了。Ionic 提供了一套模仿原生样式的 UI 组件，配合 Capacitor 插件访问原生的相机、GPS、文件系统、蓝牙等能力。好处是你全程用 Web 前端技术栈（React/Vue/Angular），开发和维护成本最低。

**适合场景**：内部工具、MVP 验证、对极端流畅要求不高的应用。**不适合场景**：需要极致原生体验（60fps 流畅动画）的应用。

### 3.3 Tauri Mobile —— 轻量级的新秀

Tauri 是你未来需要注意的一匹黑马。移动端 Tauri 使用系统自带的 WebView（iOS 上 WKWebView、Android 上 WebView），打包体积极小，比 Electron 轻很多。目前移动端的 Tauri 还在快速迭代中，但方向非常值得关注。

## 四、桌面端开发 —— 你的桌面上到处都是 JS

### 4.1 Electron —— "我也能写桌面软件了"

Electron 由 GitHub（现在是微软）创建。它的架构简单粗暴：**一个迷你的 Chromium 浏览器 + Node.js**。你的界面是一个 HTML 页面（跑在内置的 Chromium 里），你的后台逻辑由 Node.js 处理。两者可以互相通信。

优秀的 Electron 应用体现：
- **VS Code**—— 全球最流行的代码编辑器
- **Figma**—— 设计工具，性能虽然被骂但确实能用
- **Notion**—— 笔记和知识管理工具
- **Slack**—— 企业即时通讯
- **Discord**—— 游戏社群通讯工具
- **Postman**—— API 调试工具
- **Obsidian**—— 本地 Markdown 笔记工具
- **Linear**—— 项目管理工具

这些你每天都在用的桌面软件，内部全是 JavaScript。

**Electron 的优缺点**：
- 优点：开发快（前端直接转桌面）、跨平台（Win/Mac/Linux 都能跑）、生态完善、更新分发方便。
- 缺点：打包体积大（至少 100MB+，因为内置了一个 Chromium）、内存占用高、启动较慢。人们调侃 Electron 是"磁盘空间杀手"，但它的开发效率和跨平台能力让这些代价变得可接受。

### 4.2 Tauri —— 小而美的桌面方案

**Tauri** 是应对 Electron"臃肿"问题的新生力量。它的核心理念是：**不内置浏览器——用操作系统自带的 WebView 来渲染。**

- **前端部分**：仍然用 HTML/CSS/JS（React/Vue/Svelte 都行），跑在系统 WebView 中（macOS 上 WKWebView，Windows 上 WebView2，Linux 上 WebKitGTK）
- **后端部分**：用 **Rust** 编写（而非 Node.js），负责处理文件系统、网络、系统调用等
- **打包体积**：可以控制在 10MB 以下（对比 Electron 的 100MB+）
- **性能**：内存占用和 CPU 使用率都远低于 Electron

Tauri 的缺点是尚不如 Electron 生态成熟（插件、社区、Stack Overflow 答疑等），但它的成长速度非常快。到 2025 年，Tauri 2.x 已经相当稳定，很多新项目开始考虑用 Tauri 替代 Electron。

## 五、游戏开发 —— JS 不只能做表单

### 5.1 2D 游戏

- **Phaser**：最流行的 2D 网页游戏框架，社区庞大，文档丰富。物理引擎、精灵动画、音效、粒子效果应有尽有。如果你要做横版过关、RPG、塔防、休闲小游戏，Phaser 是一个很舒服的选择。
- **PixiJS**：高性能 2D 渲染引擎，底层是 WebGL，但用起来像 Canvas 一样简单。适合需要绘制大量精灵的游戏（如卡牌对战、弹幕射击）。
- **Kaboom.js**：一个非常友好的入门级游戏库，几行代码就能做出一个平台跳跃游戏的小 demo。

### 5.2 3D 游戏及 3D 渲染

- **Three.js**：最流行的 WebGL 3D 库，在 3D 图形 API 上提供了场景、相机、光照、材质、几何体等高层抽象。不管是 3D 产品展示、交互式数据可视化、VR 体验还是网页 3D 小游戏，Three.js 都能做。
- **Babylon.js**（微软）：能力比 Three.js 更全面，特别在物理引擎集成（原生支持 Havok 物理）、VR/AR、性能分析工具等方面非常强大。如果做的是重 3D 场景的应用，Babylon.js 可能更合适。
- **PlayCanvas**：基于 WebGL 的 3D 游戏引擎，有图形化编辑器，适合团队协作。

**浏览器 3D 性能怎么样？** 有了 WebGL 2 和 WebGPU，现代浏览器的 3D 性能已经足够做一些正经的游戏和产品展示了。虽然还不能跟虚幻引擎 5 比，但做个《原神》网页版……暂时洗洗睡吧。不过对于大多数需要 3D 渲染的场景，完全足够了。

## 六、AI 与机器学习 —— 新战场

### 6.1 TensorFlow.js

TensorFlow.js 是 Google 推出的 JavaScript 机器学习库，可以在浏览器或 Node.js 中**训练和运行**机器学习模型：

- **运行已有模型**：导入用 Python 训练的模型（TF.js 提供转换工具），在浏览器/Node.js 中进行推理。
- **在浏览器中训练**：用用户自己的数据进行联邦学习，模型个性化，且数据不出用户的设备（隐私友好）。
- **WebGL/WebGPU 加速**：利用 GPU 加速计算。

**应用场景**：图像识别、姿势检测（PoseNet/BlazePose）、人体分割（BodyPix）、文本分类、语音识别、推荐系统、实时人脸检测和表情识别。

### 6.2 Transformers.js —— 在浏览器里跑大模型

Transformers.js 是 HuggingFace 的 Transformers 库的 JavaScript 移植。它让你可以在浏览器（或 Node.js）里直接运行 HuggingFace 模型库里的大量预训练模型——包括文本生成、翻译、文本分类、图像分类、语音识别、图像生成（Stable Diffusion）等。

更令人兴奋的是，Transformers.js 支持 ONNX Runtime，可以在浏览器里利用 WebGPU 加速大模型的推理。这意味着用户不需要服务器、不需要 GPU 集群，就能在自己的设备上体验 AI 功能。虽然对于很大参数的模型（如完整模型的 Llama 3/DeepSeek），浏览器中运行还很有挑战，但对于参数适合的模型（ONNX Runtime Web 可在浏览器中运行许多亿级参数以下的模型），体验已经相当可用了。

### 6.3 其他值得关注的 AI 工具

- **ml5.js**：目标是"让艺术家和创意工作者更容易使用机器学习"。API 极其简洁，几行代码就能做图像分类或风格迁移。
- **Brain.js**：可以在浏览器/Node.js 里运行的神经网络库，适合教学和小规模实验，用 GPU 可以通过 `gpu.js` 加速。
- **LangChain.js**：LangChain 的 JS 版本，用于构建基于 LLM 的应用（RAG 检索增强生成、Agent 智能体、链式调用），配合矢量数据库进行语义搜索和知识库问答。

## 七、自动化与测试 —— 让 JS 替你干活

### 7.1 浏览器自动化

- **Puppeteer**（Google）：通过 DevTools Protocol 操控 Chrome/Chromium。可以做网页截图、PDF 生成、自动化表单填充、爬取 JS 渲染的页面内容、性能分析等。
- **Playwright**（微软，Puppeteer 团队核心成员出走创建的）：改良版，支持 Chromium + Firefox + WebKit 三个浏览器引擎，API 设计更现代，自动等待元素、自动管理浏览器生命周期，网络拦截能力极强。目前已经是浏览器自动化和 E2E 测试的主流工具，很多团队从 Puppeteer 迁移到了 Playwright。

**应用场景**：E2E 测试、网页截图服务、价格监控爬虫、自动化报表下载、网页性能基准测试。

### 7.2 Web 爬虫 & 数据采集

结合 `cheerio`（服务端的 jQuery 风格 HTML 解析）、`node-fetch`/`axios` 和 Puppeteer/Playwright，你可以构建一个全功能的网页数据采集系统。需要 JS 渲染的页面用 Playwright/Puppeteer 来爬，静态页面用 cheerio 来解析。

### 7.3 自动化脚本

你日常重复做的事情，很多可以用 JS 脚本自动化——批量重命名文件、处理 Excel/CSV 数据、整理下载文件夹、自动发送邮件、自动备份文件到云盘……配合 `cron` 定时任务，就可以实现全自动的周期性处理。

## 八、浏览器扩展 —— 定制你自己的浏览器

Chrome、Edge、Firefox、Safari 的浏览器扩展（Extension/Add-on）都是用 JavaScript 写的。扩展可以：

- 修改网页内容（屏蔽广告、改变排版、注入功能，如翻译、价格比对）
- 添加新的浏览器功能（标签页管理、书签同步、密码管理）
- 在后台运行（即使没有网页打开也能工作，如消息通知、定时提醒、网络请求监听）

许多你可能正在用的扩展——各类广告屏蔽器、密码管理器、翻译助手、开发者工具（React DevTools、Vue DevTools、Redux DevTools）——承载它们的核心逻辑代码就是 JavaScript。

Chrome 扩展的开发基于 Manifest V3 规范（2024 年起已经是强制的了），使用 Service Worker 做后台、`chrome.*` API 族访问浏览器功能。

## 九、小程序 —— 中国特色的"JS 宇宙"

如果你在中国做前端，"小程序"是你绕不开的一环。微信小程序、支付宝小程序、百度小程序、字节跳动小程序（抖音/头条）、快应用……中国互联网巨头各自推出了自己的"轻应用"平台，而**它们全部使用 JavaScript 作为开发语言**。

虽然每个平台的 API 和组件封装有所不同，但核心逻辑完全一样——JavaScript + 类 HTML 的模板语言 + CSS。掌握了 JS，你就掌握了所有小程序平台的通用技能。

Uni-app（uni-app）、Taro 这样的跨端框架可以让你用一套代码（Vue/React 风格）同时编译为 H5、微信小程序、支付宝小程序、字节小程序、App（React Native）等多个平台。这在商业上是巨大的效率优势。

## 十、数据可视化 —— 给数据一张漂亮的脸

### 10.1 图表库

- **ECharts**（Apache，百度捐赠）：中国最流行的数据可视化库，钉钉、淘宝数据后台随处可见它的身影。图表类型全面（折线、柱状、饼图、散点、地图、关系图、仪表盘等），中文文档和示例非常丰富。
- **Chart.js**：简单、轻量的图表库。如果只需要做几个基础的统计图表，几分钟就能搞定的封装体验非常好。
- **Highcharts**：商业库（有免费非商业许可），图表的交互和外观非常精致，适合企业级 BI 和大屏展示（金融、能源、制造业的可视化大屏很多用 Highcharts 或 ECharts）。
- **AntV**（蚂蚁集团出品）：G2（可视化图形语法）、G6（图可视化/关系图谱）、L7（地理空间数据可视化）、F2（移动端图表）、S2（多维表格）……每个库专注一个细分方向，非常专业。

### 10.2 创意与定制可视化

- **D3.js**：不是图表库，而是一个强大的**数据驱动文档操作库**。用 SVG/Canvas 或 WebGL 自定义任何你能想到的视觉表达。你可以用 D3 实现桑基图、力导向关系图、弦图、层级树图（treemap）、旭日图……数据可视化能力的天花板由你的创意决定。但学习曲线比较陡峭，随心所欲的能力背后是更多的底层细节需要自己处理。

- **Three.js**：3D 大数据可视化（比如全球航班轨迹的 3D 地球渲染、3D 分子结构展示、城市 3D 数据沙盘）。

## 十一、WebAssembly 互操作 —— JS 和"外星语言"的握手

**WebAssembly（Wasm）** 是一个低级的类汇编语言，可以在浏览器中以接近原生的速度运行。你可以把用 **C/C++、Rust、Go、Zig** 等语言写的代码编译成 Wasm，然后在网页中加载和调用。

JS 在这整个流程中扮演什么角色？**JS 是"胶水语言"**——负责加载 Wasm 模块，把 JS 对象传给 Wasm，调用 Wasm 导出的函数，接收返回结果。

应用场景：
- 图像/视频处理（如 Figma 中就是将 C++ 引擎编译成 Wasm 在浏览器里运行）
- 加密与密码学运算
- 3D 物理引擎
- SQL 数据库的浏览器运行（如 `sql.js` 是把 SQLite 编译成了 Wasm 在浏览器运行）
- 游戏引擎移植

实际上，你不需要自己写 Wasm（除非要做特别底层的优化），但你需要知道你可以调用 Wasm 来做那些 JS 自己做起来比较慢的事情。JS + Wasm 是一个互补关系，不是竞争关系。

## 十二、物联网（IoT）与嵌入式 —— 小设备里的 JS

JavaScript 甚至能跑在比手机还小的设备上：

- **Espruino**：为微控制器（STM32、ESP32、nRF52 等）设计的 JS 运行时，固件体积极小，可以直接在芯片上跑 JS。
- **Johnny-Five**：Node.js 的机器人编程框架，通过 Firmata 协议控制 Arduino 等开发板——用 JS 控制 LED 灯、马达、舵机、传感器。
- **Node-RED**（IBM 创建）：基于 Node.js 的可视化物联网流程编程工具，通过拖拽节点搭建数据流处理逻辑，广泛应用于智能家居和工业数据采集。
- **Cylon.js**：JS 的机器人和物联网框架，支持超过 40 种硬件平台。

虽然 JS 不是嵌入式开发的主流（C 语言在那里根深蒂固），但对于原型开发和创客实验来说，JS 大大降低了硬件编程的上手难度。

## 最后总结

看到这里，我希望你不会被这份"JS 能做的事情"清单吓到。它之所以能渗透到这么多领域，有一部分原因是历史巧合（谁让每台电脑都装着浏览器呢），另一部分原因是巨大的社区基数（"某个问题肯定有 JS 解决方案"的自我实现的预言）。

但你也应该知道：**没有一门语言适合所有场景。** JS 不适合写操作系统内核（这个确实是 Rust 和 C 的活），不适合做极致性能的视频编解码，不适合对 CPU 密集型计算有重负载需求的大型游戏引擎核心。

但问题是：你有多少机会去写操作系统内核？对于绝大多数软件开发者的绝大多数日常需求——Web 开发、API 服务、数据展示、工具脚本——JavaScript/TypeScript 已经能够胜任，而且效率很高。

有人把 JavaScript 比作"现代编程世界的基础设施"。这句话既是对它影响力的准确描述，也是对它无处不在的最好注脚。

所以，现在就开始写吧。不用担心你写的第一个项目是不是"最好的选择"——用任何语言开始，都比用最好的语言不动手强。
